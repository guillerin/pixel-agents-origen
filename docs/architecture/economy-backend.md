# Token Town - Economy & Backend Architecture

## 1. Token Extraction from Claude Code JSONL

### JSONL Location & Structure

Claude Code writes session transcripts as JSONL files at:
```
~/.claude/projects/<project-hash>/<session-id>.jsonl
```

Where `project-hash` = workspace path with `:`/`\`/`/` replaced by `-`.

### Token Usage Fields

Token counts live in `assistant` type records, inside `message.usage`:

```jsonc
{
  "type": "assistant",
  "message": {
    "model": "claude-opus-4-6",
    "usage": {
      "input_tokens": 3,              // Tokens sent to the model (non-cached)
      "output_tokens": 24,            // Tokens generated by the model
      "cache_creation_input_tokens": 8598,  // Tokens written to cache this request
      "cache_read_input_tokens": 6532,      // Tokens read from cache (cheaper)
      "service_tier": "standard",
      "cache_creation": {
        "ephemeral_1h_input_tokens": 0,
        "ephemeral_5m_input_tokens": 8598
      },
      "server_tool_use": {
        "web_search_requests": 0,
        "web_fetch_requests": 0
      }
    }
  }
}
```

### Key Fields for Coin Calculation

| Field | Description | Cost Weight |
|-------|-------------|-------------|
| `input_tokens` | Fresh (uncached) input tokens | 1.0x |
| `output_tokens` | Generated output tokens | 3.0x (output is ~3x more expensive) |
| `cache_creation_input_tokens` | Tokens written to cache | 1.25x (write cost + storage) |
| `cache_read_input_tokens` | Tokens read from cache | 0.1x (90% cheaper than fresh input) |

### Extraction Strategy

The existing `transcriptParser.ts` already parses every JSONL line. We add token extraction alongside the current tool tracking:

```typescript
// In processTranscriptLine(), when record.type === 'assistant':
const usage = record.message?.usage;
if (usage) {
  const tokenEvent = {
    input_tokens: usage.input_tokens ?? 0,
    output_tokens: usage.output_tokens ?? 0,
    cache_creation_input_tokens: usage.cache_creation_input_tokens ?? 0,
    cache_read_input_tokens: usage.cache_read_input_tokens ?? 0,
    model: record.message?.model ?? 'unknown',
  };
  // Send to backend for coin calculation
}
```

### Token-to-Coin Conversion Formula

```
weightedTokens = (input_tokens * 1.0)
               + (output_tokens * 3.0)
               + (cache_creation_input_tokens * 1.25)
               + (cache_read_input_tokens * 0.1)

coins = floor(weightedTokens / 1000)
```

**Rationale**: The weights approximate relative API costs. 1000 weighted tokens = 1 coin keeps the economy manageable. A typical Claude Code turn uses ~5K-50K tokens, yielding 5-50 coins per turn.

**Tuning knobs** (server-side config, adjustable without client update):
- `COIN_DIVISOR`: 1000 (tokens per coin)
- Weight multipliers per token type
- Optional daily/hourly bonus multipliers for events

## 2. Data Model

### SQLite Schema (Development) / PostgreSQL (Production)

```sql
-- Users table
CREATE TABLE users (
    id              TEXT PRIMARY KEY DEFAULT (lower(hex(randomblob(16)))),
    machine_id      TEXT UNIQUE NOT NULL,      -- VS Code vscode.env.machineId
    display_name    TEXT DEFAULT 'Agent',
    coins           INTEGER DEFAULT 0,
    total_tokens    INTEGER DEFAULT 0,         -- Lifetime weighted tokens
    total_coins_earned INTEGER DEFAULT 0,      -- Lifetime coins earned (never decreases)
    created_at      DATETIME DEFAULT CURRENT_TIMESTAMP,
    updated_at      DATETIME DEFAULT CURRENT_TIMESTAMP
);
CREATE INDEX idx_users_machine_id ON users(machine_id);

-- Coin transactions ledger (append-only)
CREATE TABLE transactions (
    id              INTEGER PRIMARY KEY AUTOINCREMENT,
    user_id         TEXT NOT NULL REFERENCES users(id),
    type            TEXT NOT NULL CHECK(type IN ('earn', 'spend')),
    amount          INTEGER NOT NULL,          -- Always positive
    balance_after   INTEGER NOT NULL,          -- Snapshot for audit
    reason          TEXT NOT NULL,             -- 'token_usage', 'purchase:{item_id}', 'refund', 'bonus'
    metadata        TEXT,                      -- JSON: session_id, model, raw token counts
    created_at      DATETIME DEFAULT CURRENT_TIMESTAMP
);
CREATE INDEX idx_transactions_user ON transactions(user_id, created_at);

-- Shop catalog (admin-managed)
CREATE TABLE shop_items (
    id              TEXT PRIMARY KEY,          -- Matches furniture-catalog.json asset id
    name            TEXT NOT NULL,
    category        TEXT NOT NULL,
    price           INTEGER NOT NULL,
    rarity          TEXT DEFAULT 'common' CHECK(rarity IN ('common', 'uncommon', 'rare', 'legendary')),
    is_available    BOOLEAN DEFAULT TRUE,
    max_per_user    INTEGER DEFAULT NULL,      -- NULL = unlimited
    created_at      DATETIME DEFAULT CURRENT_TIMESTAMP
);

-- User inventory
CREATE TABLE inventory (
    id              INTEGER PRIMARY KEY AUTOINCREMENT,
    user_id         TEXT NOT NULL REFERENCES users(id),
    item_id         TEXT NOT NULL REFERENCES shop_items(id),
    quantity        INTEGER DEFAULT 1,
    purchased_at    DATETIME DEFAULT CURRENT_TIMESTAMP,
    UNIQUE(user_id, item_id)                   -- One row per item type per user
);
CREATE INDEX idx_inventory_user ON inventory(user_id);

-- User room layouts (persisted server-side for multiplayer)
CREATE TABLE room_layouts (
    id              INTEGER PRIMARY KEY AUTOINCREMENT,
    user_id         TEXT NOT NULL REFERENCES users(id) UNIQUE,
    layout_data     TEXT NOT NULL,              -- JSON: OfficeLayout format
    updated_at      DATETIME DEFAULT CURRENT_TIMESTAMP
);
```

### Key Design Decisions

- **`machine_id`** as identity: `vscode.env.machineId` is stable per VS Code installation, no login needed.
- **Append-only transactions**: Never modify `coins` directly. All changes go through the transactions ledger, then update `users.coins` in the same transaction. This provides full audit trail.
- **`balance_after`** snapshot: Enables detecting drift between ledger sum and cached balance.
- **Inventory uses quantity**: Instead of one row per item instance, uses `UNIQUE(user_id, item_id)` with quantity. Simpler for fungible furniture items.

## 3. Server Stack

### Recommended Stack

| Layer | Choice | Rationale |
|-------|--------|-----------|
| Language | Go 1.22+ | Lightweight goroutines (~2KB each), ideal for concurrent WebSocket connections, low memory footprint |
| HTTP/WS framework | `net/http` + `gorilla/websocket` | Stdlib HTTP + battle-tested WS library. No heavy framework needed. |
| Router | `chi` | Lightweight, idiomatic Go router. No magic, composable middleware. |
| Database (dev) | SQLite via `modernc.org/sqlite` | Pure Go driver (no CGO), zero config, single file |
| Database (prod) | PostgreSQL via `pgx/v5` | Best Go PostgreSQL driver, supports pgx native protocol |
| Query layer | `sqlc` | Generates type-safe Go code from raw SQL. No ORM magic, full SQL control. |
| Migrations | `golang-migrate` | CLI + library, supports SQLite and PostgreSQL with same migration files |
| Config | `viper` or env vars | Standard Go config pattern |

### Coin Safety & Persistence

**Atomic coin operations** -- all coin changes use database transactions:

```go
// internal/economy/wallet.go
func (w *WalletService) EarnCoins(ctx context.Context, userID string, amount int, reason string, metadata map[string]any) (int, error) {
    tx, err := w.db.BeginTx(ctx, nil)
    if err != nil {
        return 0, err
    }
    defer tx.Rollback()

    var currentBalance int
    err = tx.QueryRowContext(ctx,
        "SELECT coins FROM users WHERE id = ? FOR UPDATE", userID,
    ).Scan(&currentBalance)
    if err != nil {
        return 0, err
    }

    newBalance := currentBalance + amount
    metaJSON, _ := json.Marshal(metadata)

    _, err = tx.ExecContext(ctx,
        `INSERT INTO transactions (user_id, type, amount, balance_after, reason, metadata)
         VALUES (?, 'earn', ?, ?, ?, ?)`,
        userID, amount, newBalance, reason, string(metaJSON),
    )
    if err != nil {
        return 0, err
    }

    _, err = tx.ExecContext(ctx,
        `UPDATE users SET coins = ?, total_coins_earned = total_coins_earned + ?, updated_at = CURRENT_TIMESTAMP
         WHERE id = ?`,
        newBalance, amount, userID,
    )
    if err != nil {
        return 0, err
    }

    return newBalance, tx.Commit()
}
```

**Idempotency**: Each token report includes `requestId` (from JSONL `record.requestId`). The server stores processed `requestId`s to prevent double-counting on retries.

## 4. API Design: REST + WebSocket

### REST Endpoints

```
POST   /api/auth/register          -- Register/login with machineId → returns userId + token
GET    /api/auth/me                -- Current user profile (coins, stats)

POST   /api/economy/report-tokens  -- Extension reports token usage from JSONL
GET    /api/economy/balance        -- Current coin balance

GET    /api/shop/catalog           -- All available items with prices
POST   /api/shop/purchase          -- Buy item: { itemId, quantity? }

GET    /api/inventory              -- User's owned items
GET    /api/inventory/:itemId      -- Check if user owns specific item

GET    /api/leaderboard            -- Top users by total_coins_earned
GET    /api/leaderboard/me         -- User's rank

GET    /api/rooms/:userId          -- Get a user's room layout
PUT    /api/rooms/me               -- Save current user's room layout
```

### WebSocket Events

**Connection**: `ws://server/ws` (machineId va en header `Authorization: Bearer <token>`, no en query string)

#### Server → Client Events

```typescript
// Real-time coin updates (after token report processed)
{ event: 'coins:updated', data: { coins: number, earned: number, reason: string } }

// Another player entered/left the shared view
{ event: 'player:joined', data: { userId: string, displayName: string } }
{ event: 'player:left', data: { userId: string } }

// Another player's agent status changed (for multiplayer room view)
{ event: 'agent:status', data: { userId: string, agentId: number, status: string } }

// Room layout changed by another player (for visitors)
{ event: 'room:updated', data: { userId: string, layout: OfficeLayout } }

// Shop event: new item available, limited-time sale
{ event: 'shop:announcement', data: { itemId: string, message: string } }
```

#### Client → Server Events

```typescript
// Report token usage (alternative to REST for lower latency)
{ event: 'tokens:report', data: { tokens: TokenUsage, requestId: string, sessionId: string } }

// Agent status broadcast (for multiplayer visibility)
{ event: 'agent:status', data: { agentId: number, status: string, toolName?: string } }

// Room layout update (debounced by client)
{ event: 'room:save', data: { layout: OfficeLayout } }
```

### REST vs WebSocket Decision Matrix

| Use Case | Transport | Rationale |
|----------|-----------|-----------|
| Token reporting | REST (POST) | Reliability > latency; needs idempotency guarantees |
| Coin balance updates | WS push | Real-time feedback when coins arrive |
| Shop catalog | REST (GET) | Cacheable, infrequent |
| Purchase | REST (POST) | Transactional, needs response confirmation |
| Leaderboard | REST (GET) | Cacheable, polling every 30s is fine |
| Agent status | WS | High frequency, ephemeral |
| Room layout save | REST (PUT) | Needs confirmation, debounced |
| Room layout push | WS | Real-time for visitors |

## 5. Security

### User Identity Without Login

**Primary**: `vscode.env.machineId` -- a stable UUID generated per VS Code installation. Available synchronously in the extension, no user action required.

**Flow**:
1. Extension reads `vscode.env.machineId` on activation
2. Sends `POST /api/auth/register` with `{ machineId }`
3. Server creates user if new, returns `{ userId, sessionToken }`
4. Extension stores `sessionToken` in `vscode.SecretStorage` (encrypted per-machine)
5. All subsequent requests include `Authorization: Bearer <sessionToken>`

**Session token**: HMAC-SHA256 signed JWT-like token (Go crypto/hmac + crypto/sha256), includes userId + machineId + issuedAt. Validated server-side without DB lookup. Expires after 30 days, auto-refreshed on use.

### Anti-Cheat for Token Counting

**Threat model**: A malicious user could forge JSONL lines or replay token reports to inflate their coin balance.

**Mitigations (layered)**:

1. **Request ID deduplication**: Each assistant message in JSONL has a unique `requestId` (from Anthropic API). Server tracks processed IDs. Replays are silently ignored.

2. **Rate limiting**: Max coins per hour per user (e.g., 10,000 coins/hour). Based on realistic heavy usage (~200K tokens/hour at heaviest). Alerts on anomalies.

3. **Model-aware validation**: Server knows expected token ranges per model. A `claude-haiku-4-5` response claiming 500K output tokens is flagged.

4. **Session binding**: Token reports must include a valid `sessionId` that maps to a JSONL file the server has seen activity from. The extension reports session starts.

5. **Statistical anomaly detection** (future): Compare user's token patterns against population. Flag users consistently above p99.

**What we deliberately do NOT do**:
- Server-side JSONL verification (would require API key access, privacy violation)
- Client-side obfuscation (security through obscurity doesn't work)
- Blockchain/signed receipts (over-engineering for a gamification feature)

**Pragmatic stance**: This is a fun gamification layer, not a financial system. Perfect anti-cheat is impossible without server-side API integration. The mitigations above catch casual abuse. Determined cheaters who modify their local JSONL files are "only cheating themselves" -- their inflated leaderboard position is the extent of the damage.

### Additional Security Measures

- **CORS**: Restrict to VS Code extension origin
- **Helmet**: Standard security headers on all responses
- **Input validation**: Validated in handler functions using Go structs + `json.Unmarshal`
- **SQL injection**: Prevented by `sqlc` parameterized queries (no raw string concatenation)
- **Rate limiting**: In-memory token bucket per machine_id (not per IP — multiple users can share NAT)

### Go-specific Security Notes

- Use `crypto/hmac` + `crypto/sha256` from stdlib (no external JWT library needed)
- `modernc.org/sqlite` uses parameterized queries natively — no SQL injection risk
- Rate limiting implemented via in-memory `sync.Map` + token bucket algorithm (no external dep needed for single-node; switch to Redis rate limiter for multi-node)

## 6. Pricing the Shop

### Price Tiers (coins)

| Rarity | Price Range | Examples |
|--------|-------------|---------|
| Common | 50-200 | Coffee mug, small plant, paper, book |
| Uncommon | 300-800 | Chairs, small tables, printer, bins |
| Rare | 1,000-3,000 | Desks, bookshelves, monitors, vending machine |
| Legendary | 5,000-15,000 | Server rack, full computer setup, large tables |

### Earning Rate Estimation

- Light usage (~5K tokens/turn, 20 turns/day): ~100-300 coins/day
- Medium usage (~20K tokens/turn, 50 turns/day): ~1,000-3,000 coins/day
- Heavy usage (~50K tokens/turn, 100 turns/day): ~5,000-15,000 coins/day

A common item (coffee mug, 100 coins) should be buyable within the first hour. A legendary item (server rack, 10,000 coins) should take a medium user about a week.

### Mapping furniture-catalog.json to shop_items

The existing `furniture-catalog.json` has 100 assets across 7 categories. Each asset becomes a shop item. Price is derived from category + size:

```go
// internal/economy/shop.go
func DerivePrice(asset CatalogEntry) int {
    sizeMultiplier := asset.FootprintW * asset.FootprintH
    categoryBase := map[string]int{
        "decor": 50, "misc": 75, "chairs": 100, "wall": 100,
        "storage": 200, "desks": 300, "electronics": 400,
    }
    base, ok := categoryBase[asset.Category]
    if !ok {
        base = 100
    }
    multiplier := sizeMultiplier - 1
    if multiplier < 1 {
        multiplier = 1
    }
    return base * multiplier
}
```

## 7. Integration with Extension

### Data Flow

```
JSONL file → fileWatcher.ts (reads lines)
           → transcriptParser.ts (parses, extracts usage)
           → extension sends POST /api/economy/report-tokens
           → server calculates coins, stores transaction
           → server pushes WS event coins:updated
           → extension receives, updates webview UI (coin counter)
```

### Extension Changes Needed

1. **New file**: `src/economyClient.ts` -- HTTP client for server API + WS connection
2. **Modify**: `src/transcriptParser.ts` -- extract `message.usage` from assistant records
3. **New webview message**: `coinsUpdated` with `{ coins, earned }`
4. **New UI component**: Coin counter in the webview toolbar
5. **New settings**: Server URL configuration in VS Code settings

### Offline Resilience

If the server is unreachable:
- Token reports are queued locally (up to 1000 entries in `globalState`)
- Queue is drained when connection is restored
- Offline mode still shows the office, just without coin updates or shop
- Layout changes save locally as they do today
